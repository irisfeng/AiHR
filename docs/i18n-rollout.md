# AIHR 中文国际化评估与落地建议

## 目标

AIHR 应做成：

- 中文默认
- 英文可切换
- 技术底层继续保留英文 `fieldname` / API 参数 / 状态源值

这能同时满足：

- 国内 HR 团队的可用性
- 后续中英双语产品化
- Agent、接口和升级兼容性

## 当前问题分层

### 1. 元数据层

当前自定义字段、DocType 标签和状态词大量仍是英文源文案，例如：

- `Hiring Priority`
- `Work Mode`
- `Match Score`
- `Interview Mode`
- `Ready for Review`

这是当前“每个功能页都没有中文”的最直接来源。

### 2. 客户端交互层

当前关键按钮大多已经是中文，但客户端 JS 里仍存在两种情况：

- 已硬编码中文
- 未接入正式翻译函数

这意味着：

- 中文演示可用
- 真正的中英切换还不可控

### 3. AI 输出层

AI 摘要、风险点、建议追问问题已是中文优先，但还没有标准化 `lang/locale` 参数。

### 4. 默认语言层

如果 `System Settings.language` 或 `User.language` 仍是英文，翻译资源即使存在，也可能不生效。

## 推荐落地顺序

### P0：先解决“页面看起来不是中文”

这一层应优先完成：

- 建立 `aihr/translations/zh.csv`
- 覆盖自定义字段标签
- 覆盖自定义状态词
- 覆盖 `AI Screening` DocType 关键标签
- 默认语言改为 `zh`

这一层完成后，用户最直观感受到的字段英文会明显减少。

### P1：让双语真正可切换

- 将客户端 JS 中的用户可见字符串改成可翻译写法
- 统一按钮、空态、提示语、弹窗文案
- 增加英文模式回归验证

### P2：让 AI 输出真正国际化

- API 增加 `lang/locale`
- AI 摘要、JD、代理发布包、面试资料包按语言输出
- 根据用户语言动态切换输出文案

## 重要原则

### 保留英文源值

以下内容尽量不要直接改成中文存储：

- `fieldname`
- DocType 技术名
- API 参数
- Select 状态源值

推荐做法是：

- 数据库存英文
- UI 显示中文

### 中文优先不是纯中文单语

AIHR 的正确路线不是把底层全部改成中文，而是：

- 中文默认
- 英文兼容
- 显示层翻译
- 技术层稳定
